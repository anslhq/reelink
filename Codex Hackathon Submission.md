# Proposal 1: ProofStack, the recommended build

## One-line pitch

**ProofStack lets an agent answer questions across documents, voice recordings, and video, but every sentence must be backed by a clickable page, timestamp, or frame.**

## Demo story

Use a mixed corpus that feels real but is safe to publish. For example:

1. A PDF pitch deck or product brief  
2. A voice recording or meeting transcript  
3. A short product demo video  
4. A few screenshots or docs

Then ask:

> “What are the top three customer pain points, where exactly were they mentioned, and what follow-up action should we take?”

The answer should show:

- A concise answer
- Source cards beside each claim
- Clickable PDF page references
- Audio/video jump-to-timestamp links
- A grounding score
- A warning when the system lacks enough evidence
- Optional: contradictions across sources, for example “the deck says launch is next month, but the call says the date slipped”

## Why this is likely your best shot

It combines **Multimodal Intelligence** and **Building Evals**, two of the official build directions. It is also tied to your application story, because you already talked about grounding across Plaud recordings and mixed data. The project gives you a very credible “I know this problem deeply” narrative.

The winning twist is the **evidence unit model**:

```text
EvidenceUnit {
  source_id
  modality: document | audio | video | image
  text
  start_time
  end_time
  page_number
  frame_path
  speaker
  confidence
  embedding
}
```

That turns the build from “RAG over files” into “a structured evidence compiler for multimodal agents.”

## Core architecture

```text
Upload sources
   ↓
Codex parser swarm
   ├─ Document parser: PDF/DOCX/PPTX to pages, tables, chunks
   ├─ Audio parser: transcript, speaker, timestamp
   ├─ Video parser: scenes, keyframes, transcript, visual captions
   └─ Image parser: OCR/caption/visual details
   ↓
EvidenceUnit store
   ↓
Hybrid retrieval
   ↓
Answer generator
   ↓
Grounding verifier
   ↓
Evidence UI with clickable citations
```

## Suggested stack

Use a simple, reliable stack rather than a giant framework.

Do not stop at “ask questions over files.” Add these three features:

### 1. Citation-first answers

Every generated claim must map back to an EvidenceUnit. If a sentence has no support, the UI marks it as unsupported or removes it.

## One-line pitch

**A UX layer for supervising multiple Codex agents without losing control.**

## Problem

Parallel agents are powerful, but the UX problem is real:

- Which agent is doing what?
- Which branch is safest?
- Which changes are risky?
- What can I approve, reject, or merge?
- What happened while I was away?

## What it shows

A dashboard with:

```text
Agent 1: implementing auth fix
Agent 2: writing tests
Agent 3: reviewing security risks
Agent 4: updating docs
```

Each agent has:

- Plan
- Current files touched
- Risk level
- Test status
- Diff preview
- Merge recommendation

## Why it fits

This maps directly to **UX for Agentic Applications**. It is not another chat interface. It is a control plane for agentic work.

OpenAI’s Codex materials describe Skills as playbooks Codex can follow, and plugins as ways to connect Codex to tools or sources of information. A good Control Room could expose those as repeatable workflows instead of one-off prompts. ([OpenAI](https://openai.com/academy/codex-plugins-and-skills?utm_source=chatgpt.com))

# Project 6: Ansl Work Autopilot

## One-line pitch

**A domain agent that turns emails, calls, and meetings into completed business operations, with citations and approvals.**

This is closest to your startup.

Example flow:

```text
Input:
- Email thread
- Meeting recording
- CRM notes

Output:
- Follow-up email
- Calendar hold
- CRM update
- Owner assignments
- Deadline reminders
- Evidence for every action
```

## Why it is strong

It fits **Domain Agents** very well. It also tells a commercial story.

## Why it is risky

The hackathon rules say personal projects can be extended only if the extension is significant, substantial, in a new repo, and clearly explained. Because Ansl AI already exists, judges may wonder what was built during the hackathon.

So I would use this only if the day’s theme clearly favors domain agents.

Do not pre-build the app if the rules say all work must be done on-site. But prepare the thinking.

Bring:

   Example: mini CRM, todo app, invoice app, booking form.

2. Three sample bugs  
   Example:
   - Save button says success but data is not persisted.
   - Search filter ignores partial matches.
   - Form validation accepts invalid email.

3. A 2-minute demo script.

4. A clean architecture sketch.

5. A list of Codex task prompts.

6. A plan for parallel worktrees.

7. A fallback path where the system generates the test and shows a proposed patch even if full automation is flaky.

## Best on-day build plan for Bug2Eval

### First 30 minutes

Create fresh repo.

Build or fork a tiny intentionally broken app.

Write README immediately:

```text
Bug2Eval turns vague bug reports into reproducible evals and verified Codex patches.
Built at OpenAI Codex Hackathon Sydney.
```

### Hour 1 to 2

Build the upload flow and bug parser.

Input:

- Screen recording
- Optional user text
- Repo path

Output:

- Bug summary
- Steps to reproduce
- Expected behavior
- Actual behavior

### Hour 2 to 3

Generate Playwright test.

The test should fail against the broken app.

### Hour 3 to 4.5

Wire Codex patching.

Simplest version:

```text
1. Create git worktree
2. Send bug summary and failing test to Codex
3. Codex edits code
4. Run test
5. Capture result
```

### Hour 4.5 to 5.5

Build UI:

- Bug summary
- Generated test
- Failing result
- Codex patch
- Passing result
- Saved eval

### Final hour

Record demo video and polish README.

The broad Bug2Eval concept is:

> Upload a screen recording, generate a bug report, generate a Playwright test, get Codex to fix it.

The **winning version** is more specific:

> When a bug only exists for 100 to 500 milliseconds, FrameTrace captures the video, DOM state, network state, console logs, performance timeline, React component/source mapping, and Playwright trace at the exact bad frame, then gives Codex the evidence needed to fix it.

- A screenshot happens too late.
- The model cannot see the transient visual state.

## 4. Visual testing and why motion bugs are underserved

This is probably your strongest “why now” argument.

Modern visual regression tools are usually optimized for **stable screenshots**, not validating motion itself.

Playwright screenshot assertions disable animations by default: finite animations are fast-forwarded to completion, and infinite animations are canceled to their initial state. ([Playwright](https://playwright.dev/docs/next/api/class-pageassertions?utm_source=chatgpt.com)) Chromatic similarly says it pauses CSS animations, transitions, SVG animations, and videos because consistent animation painting down to the millisecond is not practical, and it recommends toggling animation libraries like Framer Motion off for tests. ([Chromatic](https://www.chromatic.com/docs/animations?utm_source=chatgpt.com)) Percy is also centered around screenshot snapshots and baseline comparisons. ([BrowserStack](https://www.browserstack.com/docs/percy/overview/visual-testing-basics?utm_source=chatgpt.com))

That gives you a very strong wedge:

> “The current visual testing stack often disables the thing we need to test: motion.”

FrameTrace should be presented as a **motion regression eval**, not a screenshot regression tool.

# Technical architecture

## 1. Recorder

Use Playwright as the base because it can record video, create traces, capture screenshots, and generate tests. The trace viewer includes actions, screenshots/filmstrip, DOM snapshots, source code location, console, and network data. ([Playwright](https://playwright.dev/docs/next/trace-viewer?utm_source=chatgpt.com)) Playwright MCP also supports tracing and shows timeline, DOM snapshots, screenshots, network, console, and timing. ([Playwright](https://playwright.dev/mcp/tools/tracing?utm_source=chatgpt.com))

Recorder responsibilities:

```text
- Launch browser
- Start Playwright trace
- Start video recording
- Capture console logs
- Capture network requests
- Inject performance observers
- Run manual or scripted reproduction
- Stop trace/video
- Save evidence folder
```

The reproduction can be one of two modes:

### Mode A: Manual record

User opens the browser, reproduces the bug, stops recording.

This is easiest and matches your transcript.

### Mode B: Scripted record

User provides steps or Playwright code, and FrameTrace runs it.

This is better for eval generation.

---

## 2. Runtime instrumentation

Inject a small script into the page before recording:

```js
window.__frametrace = {
  layoutShifts: [],
  longAnimationFrames: [],
  clicks: [],
  routeChanges: [],
  mutations: []
}
```

Capture:

- layout shifts
- long animation frames
- click events
- route changes
- visible DOM mutations
- maybe bounding boxes for selected elements

The Layout Instability API reports layout shifts when visible viewport elements change position between frames, which is useful for detecting jank and CLS-like issues. ([MDN Web Docs](https://mdn2.netlify.app/en-US/docs/Web/API/Layout_Instability_API?utm_source=chatgpt.com)) The Long Animation Frames API, shipped in Chrome 123, helps identify slow animation frames and UI jank, and can expose script attribution details such as source URL and function information. ([Chrome for Developers](https://developer.chrome.com/docs/web-platform/long-animation-frames?utm_source=chatgpt.com))

For view transitions specifically, the View Transition API works by taking snapshots of old and new states and creating a transition pseudo-element tree. ([Chrome for Developers](https://developer.chrome.com/docs/web-platform/view-transitions/same-document?utm_source=chatgpt.com)) That matters because FrameTrace can frame itself as a tool for debugging the browser’s new transition model, not just ordinary CSS.

---

## 3. Video/frame analyzer

There are three levels.

### Level 1: Manual marker

User marks the bad timestamp in the UI.

This is reliable and enough for the hackathon.

### Level 2: Simple deterministic detection

Sample frames around transitions and look for:

- sudden brightness/background flash
- frame-to-frame pixel delta spikes
- unexpected overlay
- text overlap
- abnormal layout shift

Possible tools:

- FFmpeg for frame extraction
- pixelmatch for image diffing
- SSIM for perceptual similarity
- OpenCV optical flow for motion changes

OpenCV’s optical flow estimates object movement between consecutive frames, which is useful if you want to detect weird motion or warped transitions. ([OpenCV Documentation](https://docs.opencv.org/4.x/d4/dee/tutorial_optical_flow.html?utm_source=chatgpt.com)) Pixelmatch can compare images at the pixel level and is commonly used in screenshot testing. ([GitHub](https://github.com/mapbox/pixelmatch?utm_source=chatgpt.com)) SSIM is also useful because plain MSE does not always match perceived similarity. ([Scikit-Image](https://scikit-image.org/docs/0.24.x/auto_examples/transform/plot_ssim.html?utm_source=chatgpt.com))

### Level 3: Gemini video understanding

Use Gemini only as an optional enhancement.

Gemini’s video understanding docs say it can process videos, answer questions about video content, refer to timestamps, and ingest video via upload, inline video, or YouTube URLs. The File API stores video at 1 FPS by default and adds timestamps every second. ([Google AI for Developers](https://ai.google.dev/gemini-api/docs/video-understanding?utm_source=chatgpt.com))

This is helpful, but do not make the whole project dependent on Gemini. Your product should be model-agnostic.

A good use:

```text
Ask Gemini:
"At what timestamp does the UI flash, overlap, or visually glitch?
Return only JSON with timestamp, issue_type, description."
```

Then FrameTrace maps that timestamp to runtime evidence.

# Why this can win

This project has a stronger winning shape than pure ProofStack or generic Bug2Eval because it has:

1. **A painful, specific problem**  
   Motion bugs are hard to capture, explain, and test.

2. **A strong technical wedge**  
   Video timestamp plus DOM plus Playwright trace plus performance plus source context.

3. **A clear Codex role**  
   Codex does not just help build the app. Codex consumes the generated evidence pack and fixes the bug.

4. **A visual demo**  
   Judges can see the bug, see the bad frame, see the generated eval, and see the fix.

5. **A credible future**  
   This can become an MCP, a Chrome extension, a CI tool, or an open-source motion regression framework.

6. **A reason existing tools do not fully solve it**  
   Session replay captures what happened. Bug reporting tools create tickets. Visual testing often freezes animations. FrameTrace turns the motion bug itself into an eval for coding agents.

# 5. Why MCP is required

You are right: without MCP, the coding harness cannot reliably use the video.

# 12. Actual model plan

## OpenAI-first model path

Use OpenAI for image/keyframe reasoning:

1. Extract keyframes with FFmpeg.
2. Create contact sheets or before/bad/after frame groups.
3. Send frames to a vision-capable OpenAI model through image input.
4. Ask for strict JSON:
   - issue type
   - timestamp
   - visible symptoms
   - likely repro steps
   - confidence
5. Use Codex for source/code reasoning and patching.

This avoids depending on raw video ingestion. OpenAI’s public image/vision docs support image inputs via Responses API, but the relevant docs list image formats, not MP4 video input for analysis, so frame sampling is the safer OpenAI-native implementation. ([OpenAI Platform](https://platform.openai.com/docs/guides/images-vision?utm_source=chatgpt.com))

## Optional Gemini path

Use Gemini only for direct MP4 timestamping:

```text
Input: video
Prompt: “Find the timestamp where the UI visibly breaks. Return JSON only.”
Output: { timestampMs, issueType, description, confidence }
```

Gemini’s video understanding docs say Gemini can process videos, answer questions about video content, and refer to specific timestamps. ([Google AI for Developers](https://ai.google.dev/gemini-api/docs/video-understanding?utm_source=chatgpt.com))

## Deterministic path

Use deterministic analysis first because it is cheaper and more reliable for the demo:

| Bug type | Deterministic signal |
|---|---|
| Loading flash | brightness/background delta |
| Layout shift | Layout Instability API |
| Jank | Long Animation Frames API |
| Network bug | failed/slow network request |
| Console bug | console error near timestamp |
| Button no-op | click event with no DOM/network/navigation change |
| Transition glitch | frame diff spike + overlapping bounding boxes |
| Data persistence | DOM state before/after reload |

The Layout Instability API measures visible elements changing position between frames, and the Long Animation Frames API helps identify slow UI updates and jank, including script attribution details in Chrome. ([MDN Web Docs](https://mdn2.netlify.app/en-US/docs/Web/API/Layout_Instability_API?utm_source=chatgpt.com))

## React model/context path

For React apps:

- Use React Grab/react-grep-style fiber inspection.
- Capture source file/line, component tree, selector, computed styles, bounding rect.
- Optionally add React Scan later for render performance issues.

React Scan is relevant as a future layer because it detects React performance issues, highlights components, and has a profiler for FPS drops and slow interactions. ([GitHub](https://github.com/aidenybai/react-scan?utm_source=chatgpt.com))

# 13. Timestamp-to-DOM alignment

This is one of the most important implementation details.

## For BugTrace-recorded sessions

When BugTrace records the browser session, inject a timestamp bridge:

```js
window.__BUGTRACE = {
  runId,
  recordingStartWallTime: Date.now(),
  recordingStartPerf: performance.now(),
  events: []
}
```

Log every event like:

```json
{
  "type": "click",
  "perfTimeMs": 522.4,
  "wallTimeMs": 1777355432522,
  "selector": "a[href='/blog']",
  "text": "Blog"
}
```

Extract video frames with FFmpeg including presentation timestamps.

Then map:

```text
video timestamp → recording wall time → performance.now timeline → DOM/network/React event window
```

For even easier calibration, show a tiny hidden or visible debug timer overlay during capture:

```text
data-bugtrace-time="000522ms"
```

The model/keyframe analyzer can read or correlate that time.

## For raw user-uploaded videos

You cannot truly map video to DOM/source unless you recapture or have instrumentation.

So the flow is:

```text
Raw video
  → visual analysis
  → inferred repro steps
  → agent records fresh instrumented repro
  → runtime evidence + source mapping
  → eval + fix
```

This is the right UX and the right technical truth.

# 14. PRD

## Product

**BugTrace MCP**

## Target users

- frontend engineers
- design engineers
- QA engineers
- founders using AI coding agents
- teams shipping with Codex/Cursor/Claude Code

## Problem

AI coding agents can read code and logs, but they struggle with visual/runtime bugs because bug reports are fragmented:

- screenshot loses timing
- video lacks source/runtime context
- network logs lack what user saw
- DOM snapshots lack motion
- visual testing often freezes animations
- coding agents cannot reliably consume raw videos inside coding harnesses
- generated fixes are not trustworthy unless verified by an eval

Playwright’s screenshot assertions disable animations by default, which is useful for stable visual tests but bad for debugging motion itself. ([Playwright](https://playwright.dev/docs/next/api/class-pageassertions?utm_source=chatgpt.com))

## Product goal

Turn any frontend bug report into a **BugPack** that a coding agent can understand, test, fix, and verify.

## Core user stories

### Story 1: video-first bug fixing

As a developer, I can give Codex a video file path and say “fix this,” and BugTrace creates an evidence pack with keyframes, issue summary, repro steps, and generated eval.

### Story 2: agent-run testing

As a developer, I can ask Codex to test a page, and BugTrace records the session with video, trace, console, network, DOM, performance, and React source context.

### Story 3: React source mapping

As a React developer, I can install BugTrace/React Grab and have every selected/bad UI element mapped to component, file, line, selector, computed style, and bounding box.

### Story 4: eval loop

As a developer, I can get a Playwright regression test that fails before the fix and passes after the fix.

### Story 5: verification

As a reviewer, I can open the BugPack and see before/after proof.

| Feature | Acceptance criteria |
|---|---|
| MCP server | Codex can call at least `analyze_video_file`, `record_browser_session`, `generate_eval`, `run_eval`, `get_bugpack` |

| BugPack writer | Creates `bug.md`, `evidence.json`, frames, and recording links |
| Browser recorder | Captures Playwright video, trace, console, network |

```text
screenshots lose timing
videos lose DOM/source/network context
logs lose what the user saw
browser agents do slow screenshot-DOM-click loops
Codex cannot directly reason over arbitrary local video files without a tool bridge
```

Your transcript’s key insight is that the missing product is not a screenshot tool. It is a video-first bug understanding system that then ties the bad timestamp to DOM, source, and eval context. `Pasted text.txt`

## Product goal

Let a developer say:

```text
“Fix this bug video.”
```

and have the coding agent produce:

```text
1. What went wrong
2. Where it happened in the video
3. How to reproduce it
4. What DOM/source/runtime state existed at that moment
5. A failing regression test
6. A verified fix
```

## P0 acceptance criteria

| Feature | Acceptance criteria |
|---|---|
| Raw video analysis | Given `.mp4/.mov/.webm`, model returns at least one bug timestamp and summary |
| BugPack creation | Tool creates folder with `bug.md`, `evidence.json`, frames, clips |
| MCP access | Codex can call local tools using video path |
| Runtime recapture | Given app URL and repro steps, tool records instrumented browser session |
| React source context | In demo React app, bad/selected element maps to component and file |
| Eval generation | Tool generates a Playwright test from bug type |
| Eval verification | Test fails before fix and passes after fix |
| Codex workflow | Skill tells Codex how to use the MCP tools |

## P1 acceptance criteria

| Feature | Acceptance criteria |
|---|---|
| Multi-model support | Gemini direct video, OpenAI keyframes fallback |
| Similar bug search | Embeddings over past BugPacks |
| Agent self-test mode | Codex can record its own browser testing session |
| Before/after video compare | Tool proves visual issue is gone |
| Component ranking | Source files ranked by visual/runtime relevance |

# What we build first

The minimum valuable prototype is:

```text
MCP tool accepts video path
  → Gemini analyzes video
  → returns bug timestamp and repro steps
  → MCP extracts evidence around timestamp
  → if URL/repo available, records instrumented reproduction
  → React context maps UI to source file
  → generates Playwright eval
  → Codex fixes and verifies
```

No full dashboard needed. No complete Chrome extension needed. No video embeddings needed.

## Day-before build spike

Since you are doing prep before the hackathon, the spike should test only the risky parts:

1. **Gemini raw video call**  
   Upload your website bug video. Confirm JSON timestamp quality.

2. **MCP local video path tool**  
   Basic TypeScript MCP server with `analyze_video_file(videoPath)`.

3. **React context extraction**  
   Try React Grab/react-grep on your site or a demo Next app and capture file/component/selector.

4. **Instrumented replay**  
   Use Playwright only to recapture the same route/action and save trace/video/logs.

5. **Generated eval**  
   Generate one Playwright test for one bug type.

That is the product skeleton.

# Correct build priority

## P0: video-first bug intelligence

This must work.

```text
videoPath -> Gemini -> JSON bug timestamp
```

Without this, the product collapses back into FFmpeg/frame diff.

## P1: MCP bridge

This must work.

```text
Codex -> MCP -> local video path -> BugPack
```

Without this, the workflow is still “paste into Gemini, copy result into Codex.”

## P2: timestamp to context

This is the product depth.

```text
bad timestamp -> frames + DOM + React source + network + console
```

## P3: generated eval

This is the winning hackathon artifact.

```text
BugPack -> failing Playwright test
```

## P4: Codex fix verification

This is the closing loop.

```text
Codex patch -> eval passes -> before/after proof
```

# Example end-to-end demo

## User says

```text
Use the bug tool to fix /Users/harsha/Desktop/transition-bug.mov.
The app is running at http://localhost:3000.
Repo is current directory.
```

## Codex calls MCP

```json
{
  "tool": "analyze_video_file",
  "videoPath": "/Users/harsha/Desktop/transition-bug.mov",
  "repoRoot": ".",
  "appUrl": "http://localhost:3000",
  "hint": "View transition issue"
}
```

## MCP returns

```json
{
  "runId": "run_001",
  "bugFound": true,
  "summary": "The heading text overlaps during route transition to /blog.",
  "issues": [
    {
      "issueType": "route_transition_glitch",
      "startMs": 430,
      "endMs": 720,
      "confidence": 0.91,
      "actual": "Outgoing and incoming heading text overlap.",
      "expected": "Only one heading should be readable during transition.",
      "reproSteps": [
        "Open home page",
        "Click Blog",
        "Observe heading during transition"
      ]
    }
  ],
  "bugpackPath": ".bugtrace/run_001"
}
```

## Codex calls

```json
{
  "tool": "record_instrumented_session",
  "appUrl": "http://localhost:3000",
  "repoRoot": ".",
  "steps": ["click a[href='/blog']"],
  "enableReactContext": true
}
```

## MCP maps context

```json
{
  "tool": "map_timestamp_to_context",
  "runId": "run_001",
  "timestampMs": 520
}
```

## MCP returns

```json
{
  "likelySourceFiles": [
    "app/page.tsx",
    "app/blog/page.tsx",
    "components/motion/page-transition.tsx"
  ],
  "reactContext": [
    {
      "selector": "main h1",
      "component": "HeroTitle",
      "file": "components/hero-title.tsx",
      "line": 18
    }
  ],
  "visualFrames": {
    "before": ".bugtrace/run_001/frames/before.png",
    "bad": ".bugtrace/run_001/frames/bad.png",
    "after": ".bugtrace/run_001/frames/after.png"
  }
}
```

## MCP generates eval

```json
{
  "tool": "generate_regression_eval",
  "runId": "run_001",
  "issueId": "issue_1",
  "evalType": "motion"
}
```

## Codex patches and verifies

```text
- Opens likely source files
- Applies minimal fix
- Runs generated eval
- Captures after video
- Reports pass/fail
```
